# Internationalisation (i18n) Guide

## Adding a New Language

To add support for a new language (e.g., French, German, Portuguese):

### 1. Register the locale in `AppLocale`

Edit `lib/core/config/locale_config.dart` and add a new value to the `AppLocale` enum:

```dart
enum AppLocale {
  en(Locale('en'), 'English'),
  es(Locale('es'), 'Español'),
  fr(Locale('fr'), 'Français'),  // <-- add new locale
  // ...
}
```

This is the **single source of truth**. All other registrations derive from this enum:

| Registration | Derives from |
|---|---|
| `l10n.yaml` `supported-locales` | Must be manually kept in sync (Flutter limitation) |
| `main.dart` `supportedLocales` | `AppLocale.supportedLocales` (automatic) |
| `profile_screen.dart` dropdown | `AppLocale.buildDropdownItems()` (automatic) |
| `localeProvider` default | `AppLocale.supportedLocales` (via generated code) |

### 2. Create a new ARB file

Copy `lib/l10n/app_en.arb` to `lib/l10n/app_XX.arb` (where `XX` is the language code, e.g., `fr`, `de`, `pt`).

Replace the `"@@locale"` value at the top and translate all strings.

### 3. Update `l10n.yaml`

Add the new locale to the `supported-locales` list in `l10n.yaml`:

```yaml
supported-locales:
  - en
  - es
  - xx  # <-- add new locale
```

### 4. Add localised language name

Add a new localised key for the language name in all ARB files (e.g., `"french": "French"` in `app_en.arb`, `"french": "Français"` in `app_es.arb`, etc.).

### 5. Regenerate Dart code

Run the localization generator:

```bash
bash scripts/gen_l10n.sh
```

Or directly:

```bash
flutter gen-l10n
```

### 6. Add tests

1. Add test assertions for the new locale in `test/l10n/app_localizations_test.dart`.
2. The key-parity test in `test/l10n/app_localizations_coverage_test.dart` automatically verifies that every key in `app_en.arb` has a counterpart in all other locale ARB files (and vice versa). It will fail if any key is missing, ensuring full coverage across all supported locales.

### 7. Verify hardcoded strings

After adding a new locale, lint the codebase for any remaining hardcoded user-facing strings:

```bash
grep -rn "'[A-Z][a-z]" lib/features/ lib/core/ | grep -v '.arb' | grep -v 'import'
```

Any matches should be extracted to ARB keys and replaced with `AppLocalizations.of(context)!` calls.

## ARB Description Convention

All `@key` descriptions in ARB files **must be in English** (the source language), regardless of the target locale. This ensures:
- Developers who may not speak the target language can understand the context
- Generated code comments are consistent across locales
- Translator tools can reference the descriptions

## Regeneration

Run `bash scripts/gen_l10n.sh` to regenerate the Dart code from ARB files after any ARB changes.

## Register Convention (Spanish)

The Spanish translation uses **formal *usted*** register:
- Possessive adjectives: use **su/sus** (not *tu/tus*)
- Imperative verbs: use **formal *usted* conjugation** (-e/-a endings)
- This convention must be maintained in all Spanish translations.

### Translation Consistency Guidelines

1. **Same English concept → same Spanish word.** Do not translate the same English word differently in different keys (e.g., "Accuracy" must be the same word everywhere).
2. **Positive framing.** Prefer constructive language (e.g., "Áreas por mejorar" over "Áreas débiles").
3. **Natural nouns.** When a word functions as a noun (e.g., "Weak" as a category label), use natural Spanish nouns rather than direct adjective translations (e.g., "Por mejorar" over "Débil").

## Regional Variant Infrastructure

### Fallback Chain

The `AppLocale.resolveLocale()` method in `lib/core/config/locale_config.dart` implements the locale resolution callback used in `main.dart`. When a user's device locale is a regional variant (e.g., `es_MX`, `es_ES`, `es_AR`), the resolver maps it to the base locale by matching on `languageCode`:

```
es_MX → es (neutral Latin American Spanish)
es_ES → es (base Spanish ARB)
es_AR → es (base Spanish ARB)
```

### Adding a Regional Variant

To add a regional variant (e.g., Spain-specific Spanish `es_ES`):

1. Create `app_es_ES.arb` with only the keys that differ from `app_es.arb` (Flutter merges parent locale overrides automatically).
2. Update `AppLocale` enum if the variant needs its own `Locale` object in `supportedLocales`.
3. The `resolveLocale` callback automatically handles the fallback: `Locale('es_ES')` resolves to `Locale('es')` if `es_ES` is not in the supported list, using the base ES ARB as fallback.

### Documented Convention

- `'es'` targets neutral Latin American Spanish (formal "usted" register).
- Regional variants (`es-MX`, `es-ES`, `es-AR`) are not yet supported but infrastructure is ready.
- To add Spain-specific vocabulary, create `app_es_ES.arb` with overrides for terms like "ordenador", "vale", "añadir", etc.

## Duration Abbreviation Policy

Time abbreviations follow a **locale-sensitive** policy:

| Unit | EN | ES |
|---|---|---|
| Seconds | `1s` / `{count}s` | `1s` / `{count}s` |
| Minutes | `1m` / `{count}m` | `1min` / `{count}min` |
| Hours | `1h` / `{count}h` | `1h` / `{count}h` |
| Days | `1d` / `{count}d` | `1d` / `{count}d` |

Minutes are locale-sensitive (`m` in EN, `min` in ES) to avoid ambiguity in Spanish where "m" could be confused with "metro" (meter). Other units remain universal.

## CI Validation

A validation script is available at `scripts/validate_arb_no_duplicates.dart` to check for duplicate translation keys in ARB files. Run it before committing ARB changes:

```bash
dart run scripts/validate_arb_no_duplicates.dart
```

This should be integrated into your CI pipeline or pre-commit hook. Duplicate keys cause silent translation degradation because JSON takes the last value.

## PR Review Checklist

When reviewing a PR that touches `app_es.arb` or any generated Spanish localization:
- [ ] All new or changed Spanish strings use formal **usted** register
- [ ] No `tú`, `tu`, `tus`, `te`, or informal imperative (-a/-e endings for -ar/-er verbs) is introduced
- [ ] Possessive adjectives use **su/sus** (not *tu/tus*)
- [ ] Imperative verbs use **formal *usted* conjugation** (3rd person singular subjunctive: -e for -ar verbs, -a for -er/-ir verbs)
- [ ] No duplicate translations of the same English concept
- [ ] Positive/constructive framing preferred for weakness-related terms
- [ ] AI system prompt key (`quickGuideSystemPrompt`) exists in the new locale's ARB and instructs the model to respond in that language
- [ ] Fallback intent routing (keyword matching) is updated to include the new locale's keywords or removed in favour of a locale-agnostic approach
- [ ] All `@key` descriptions in ARB files are in English per the ARB Description Convention

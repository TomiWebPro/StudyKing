# Internationalisation (i18n) Guide

## Adding a New Language

To add support for a new language (e.g., French, German, Portuguese):

### 1. Create a new ARB file

Copy `lib/l10n/app_en.arb` to `lib/l10n/app_XX.arb` (where `XX` is the language code, e.g., `fr`, `de`, `pt`).

Replace the `"@@locale"` value at the top and translate all strings.

### 2. Register the locale in `supportedLocales`

Add the new locale to `supportedLocales` in `lib/main.dart` (`StudyKingApp.build()`):

```dart
supportedLocales: const [
  Locale('en'),
  Locale('es'),
  Locale('xx'),  // <-- add new locale
],
```

### 3. Register in locale detection and dropdown

Add the new locale to the device locale auto-detection in `lib/main.dart` (`localeProvider` initial value) and to the language dropdown in `lib/features/settings/presentation/profile_screen.dart`.

### 4. Regenerate Dart code

Run the localization generator:

```bash
bash scripts/gen_l10n.sh
```

Or directly:

```bash
flutter gen-l10n
```

### 5. Register in the language dropdown

Add the new language option to `lib/features/settings/presentation/profile_screen.dart`:

```dart
DropdownMenuItem(value: 'xx', child: Text(l10n.XXLanguageName)),
```

Add a new localised key for the language name in all ARB files (e.g., `"french": "French"` / `"french": "Français"`).

### 6. Add tests

1. Add test assertions for the new locale in `test/l10n/app_localizations_test.dart`.
2. The key-parity test in `test/l10n/app_localizations_coverage_test.dart` automatically verifies that every key in `app_en.arb` has a counterpart in all other locale ARB files (and vice versa). It will fail if any key is missing, ensuring full coverage across all supported locales.

### 7. Verify hardcoded strings

After adding a new locale, lint the codebase for any remaining hardcoded user-facing strings:

```bash
grep -rn "'[A-Z][a-z]" lib/features/ lib/core/ | grep -v '.arb' | grep -v 'import'
```

Any matches should be extracted to ARB keys and replaced with `AppLocalizations.of(context)!` calls.

## Regeneration

Run `bash scripts/gen_l10n.sh` to regenerate the Dart code from ARB files after any ARB changes.

## Register Convention (Spanish)

The Spanish translation uses **formal *usted*** register:
- Possessive adjectives: use **su/sus** (not *tu/tus*)
- Imperative verbs: use **formal *usted* conjugation** (-e/-a endings)
- This convention must be maintained in all Spanish translations.

### Translation Consistency Guidelines

1. **Same English concept → same Spanish word.** Do not translate the same English word differently in different keys (e.g., "Accuracy" must be the same word everywhere).
2. **Positive framing.** Prefer constructive language (e.g., "Áreas por mejorar" over "Áreas débiles").
3. **Natural nouns.** When a word functions as a noun (e.g., "Weak" as a category label), use natural Spanish nouns rather than direct adjective translations (e.g., "Por mejorar" over "Débil").

### CI Validation

A validation script is available at `scripts/validate_arb_no_duplicates.dart` to check for duplicate translation keys in ARB files. Run it before committing ARB changes:

```bash
dart run scripts/validate_arb_no_duplicates.dart
```

This should be integrated into your CI pipeline or pre-commit hook. Duplicate keys cause silent translation degradation because JSON takes the last value.

### PR Review Checklist

When reviewing a PR that touches `app_es.arb` or any generated Spanish localization:
- [ ] All new or changed Spanish strings use formal **usted** register
- [ ] No `tú`, `tu`, `tus`, `te`, or informal imperative (-a/-e endings for -ar/-er verbs) is introduced
- [ ] Possessive adjectives use **su/sus** (not *tu/tus*)
- [ ] Imperative verbs use **formal *usted* conjugation** (3rd person singular subjunctive: -e for -ar verbs, -a for -er/-ir verbs)
- [ ] No duplicate translations of the same English concept
- [ ] Positive/constructive framing preferred for weakness-related terms
- [ ] AI system prompt key (`quickGuideSystemPrompt`) exists in the new locale's ARB and instructs the model to respond in that language
- [ ] Fallback intent routing (keyword matching) is updated to include the new locale's keywords or removed in favour of a locale-agnostic approach
- [ ] `pubspec.yaml` `flutter:` section lists the new locale in `supportedLocales`

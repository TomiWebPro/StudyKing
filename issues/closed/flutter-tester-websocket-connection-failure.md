# `flutter test` fails with WebSocketException on all projects

**Severity:** critical
**Affected area:** Flutter test runner / `flutter_tester` binary
**Reported by:** developer

## Description

`flutter test` fails on **every** Flutter project (not just StudyKing) with the error:

```
Failed to load "..._test.dart": Unable to connect to flutter_tester process: 
  WebSocketException: Invalid WebSocket upgrade request
```

The `flutter_tester` binary starts briefly (PID confirmed in verbose logs) but the test harness cannot establish a WebSocket connection to it. The process is terminated with exit code `-9` (SIGKILL) approximately 700–1000 ms after launch. This makes it **impossible to run any widget or unit tests** that depend on the `flutter_test` package.

## Steps to reproduce

1. Create or open any Flutter project.
2. Create a minimal test file, e.g. `test/widget_test.dart`:
   ```dart
   import 'package:flutter_test/flutter_test.dart';
   void main() {
     testWidgets('trivial test', (tester) async {
       expect(1 + 1, 2);
     });
   }
   ```
3. Run `flutter test` (or `xvfb-run flutter test` on headless systems).
4. Observe the error.

## Expected behavior

Tests should compile and run normally. The `flutter_tester` process should connect back to the test harness via the advertised WebSocket port and execute the test.

## Actual behavior

- `flutter_tester` starts but never sends the expected WebSocket upgrade response.
- After ~1 second the test harness reports the WebSocket failure and kills the process.
- No useful stderr output is emitted by `flutter_tester` (the verbose Flutter logs show no crash details).

## Code analysis

### Root cause investigation

The `flutter_tester` binary itself is functional:

```
$ /path/to/flutter_tester
[ERROR:flutter/shell/testing/tester_main.cc(586)] Dart kernel file not specified.
Exit code: 1
```

This confirms the binary can start and execute its entry point. The failure occurs when it is given a valid `.dill` kernel file and expected to connect back to the test harness via the `SERVER_PORT` environment variable over WebSocket.

Potential causes (in order of likelihood):

1. **Internal crash during Dart isolate initialization** — The `listener.dart.dill` (generated by Flutter's test infrastructure) may trigger a crash during bootstrap that happens before the WebSocket handshake completes. The crash could be silent (e.g., abort with no stderr).

2. **Fontconfig issue** — The test harness writes a temporary `fonts.conf` pointing to the `SERVER_PORT` value as part of the font configuration. A misconfiguration could cause `flutter_tester` to crash during font loading.

   Verbose log confirms:
   ```
   FONTCONFIG_FILE: /tmp/flutter_tools.XXX/flutter_test_fonts.YYY/fonts.conf
   ```

3. **Flutter 3.44.0 regression** — This version of Flutter (released 2026-05-15) may contain a regression in the test runner's WebSocket communication on Linux. The `engine` revision `4c525dac5e` could have introduced a change that broke the `flutter_tester` ↔ test harness handshake on certain kernel/library combinations.

4. **Kernel 6.17 incompatibility** — The custom kernel `6.17.0-35-generic` on this Ubuntu 24.04 installation may have changed behavior around Unix domain sockets or loopback networking that affects the WebSocket handshake.

## Suggested approach

1. **Update Flutter SDK** — Check if a newer patch of Flutter 3.44.x or a newer channel release addresses the regression. Run `flutter upgrade` and retest.

2. **Re-install Flutter engine binaries** — Delete and re-download the Linux engine artifacts:
   ```bash
   rm -rf /path/to/flutter/bin/cache/artifacts/engine/linux-x64
   flutter precache
   ```

3. **Downgrade Flutter** — If the issue is a regression in 3.44.0, pin to a known-working version (e.g., 3.32.x or earlier).

4. **Run with `--no-test-assets`** — If asset generation is the culprit, try:
   ```bash
   flutter test --no-test-assets
   ```

5. **Check fontconfig** — Inspect the generated `fonts.conf` file at `/tmp/flutter_tools.XXX/flutter_test_fonts.YYY/fonts.conf` during a test run to verify it is valid XML.

6. **Report upstream** — If confirmed as a Flutter SDK regression, file a bug at https://github.com/flutter/flutter/issues with the verbose log and environment details.

## Resolution

### Root cause

The `HTTP_PROXY`/`http_proxy` environment variables were set to `http://127.0.0.1:10808/` (configured via GNOME Settings → Network → Proxy for the `xray` proxy at `~/.local/share/v2rayN/bin/xray/`). When `flutter_tester` starts, it inherits these proxy env vars from the parent `flutter test` process (via Dart's `Process.start(includeParentEnvironment: true)` default behavior).

The `listener.dart.dill` code running inside `flutter_tester` calls `WebSocket.connect('ws://127.0.0.1:<SERVER_PORT>')`. Dart's `HttpClient` (used by `WebSocket.connect`) sees the proxy env vars and attempts to route the WebSocket connection through the proxy at `127.0.0.1:10808`. The proxy returns HTTP 400 instead of a valid WebSocket upgrade response, causing `WebSocketException: Invalid WebSocket upgrade request`.

### Why NO_PROXY didn't help

The `no_proxy`/`NO_PROXY` environment variables contained `127.0.0.0/8` (CIDR notation), exported by GNOME's proxy `ignore-hosts` setting. **Dart's `HttpClient` does not parse CIDR notation in `NO_PROXY`** — it treats `127.0.0.0/8` as a literal hostname string, which never matches `127.0.0.1`, so the proxy is always used.

Additionally, Dart reads `http_proxy` (lowercase) for proxy configuration but reads `NO_PROXY` (uppercase) for bypass rules. Having lowercase proxy vars but uppercase no_proxy created an inconsistency.

### Fix applied

Three changes were made:

1. **Flutter SDK upgraded** from 3.44.0 → 3.44.4 (Dart 3.12.0 → 3.12.2, engine 4c525dac5e → a10d8ac38d).

2. **GNOME proxy ignore-hosts updated** to use explicit `127.0.0.1` instead of CIDR `127.0.0.0/8`:
   ```bash
   gsettings set org.gnome.system.proxy ignore-hosts "['localhost', '127.0.0.1', '::1']"
   ```

3. **`~/.bashrc` fix added** to replace `127.0.0.0/8` with `127.0.0.1` in `no_proxy`/`NO_PROXY` at shell startup, as a safety net:
   ```bash
   if [[ -n "$no_proxy" ]]; then
     export no_proxy="${no_proxy//127.0.0.0\/8/127.0.0.1}"
   fi
   if [[ -n "$NO_PROXY" ]]; then
     export NO_PROXY="${NO_PROXY//127.0.0.0\/8/127.0.0.1}"
   fi
   ```
